# Vocalype Brain — Operating Contract

Version: 1.0
Date: 2026-04-24
Status: permanent — do not remove or override

This document is the standing contract between the founder, Vocalype Brain,
and any implementation model (Claude Code, Codex, Aider) working on this repo.
Future mission prompts may reference it instead of repeating safety rules.

---

## 1. Role of the Implementation Model

- Claude / Codex / Aider is an **implementation worker**, not the founder and not the architect.
- It executes a scoped task defined by the founder or generated by Vocalype Brain.
- It does not expand scope, infer undeclared intent, or make architectural decisions on its own.
- When the task is ambiguous, it **stops and asks** rather than guessing.
- When a safety rule would be violated, it **stops and reports** rather than proceeding.
- It does not commit, deploy, or delete without explicit instruction.

---

## 2. Standard Workflow for Every Version

Every Brain version or product task follows this sequence in order.
Do not skip steps. Do not combine steps into one commit.

| Step | Action | Commit type |
|---|---|---|
| 1 | `git status --short` — confirm repo is clean before starting | — |
| 2 | Design / plan (planning_only) — write design doc to `outputs/` | `docs(brain): ...` |
| 3 | Commit design | separate commit |
| 4 | Minimal implementation — smallest code that satisfies the design | `feat(brain): ...` or `feat(app): ...` |
| 5 | Validation — compile check, lint, dry-run, smoke test | — |
| 6 | Commit implementation | separate commit |
| 7 | Real-loop validation — run the full pipeline, confirm output | — |
| 8 | Review / record results — update `results.jsonl`, `lessons_learned.md`, `wins.md` | `docs(brain): ...` |
| 9 | Commit validation result | separate commit |

**One task = one commit. Never combine brain files with product files in the same commit.**

## 3. Safety Rules

These rules are always active. They cannot be overridden by a mission prompt.

### Code boundaries
- Do **not** modify `backend/`, `src-tauri/`, `src/lib/auth/`, `src/lib/license/` unless the founder explicitly approves in the current session.
- Do **not** modify payment, billing, security, Rust runtime, or audio runtime code under any circumstances.
- Do **not** modify product code (`src/`) during a `planning_only` or `measurement_task`.
- Brain scripts (`vocalype-brain/`) never write to product files. Read-only access only.

### Commits
- Do **not** use `--no-verify` on any commit.
- Do **not** combine product code with Brain memory files in the same commit.
- Do **not** commit when `git status` shows unclear or out-of-scope changes.
- Always run `git diff --stat` before staging — formatter tools silently modify out-of-scope files.
- Clean out-of-scope formatting changes with `git restore <file>` before committing.

### Deployment and destructive actions
- Do **not** auto-deploy.
- Do **not** delete files without explicit founder instruction.
- Do **not** run `git reset --hard`, `git push --force`, or `git checkout -- .` without explicit instruction.

### Formatting
- Do **not** run broad `bun run format` or `prettier --write .` before a product commit unless the founder requests it.
- If format is run and modifies out-of-scope files, restore them before staging.

### Scope
- Do **not** widen task scope beyond the approved files.
- Do **not** add new dependencies without explicit approval.
- If the approved scope is insufficient to complete the task safely, stop and report.

## 4. Task Types

Every task must be classified before implementation begins.

| Type | Condition | Allowed output |
|---|---|---|
| `planning_only` | Safety gate fails, risk=high, vague scope, or scope too broad | Design doc, plan file — no product code |
| `measurement_task` | Data collection required before implementation | Checklist, observation plan — no product code |
| `proposal_task` | Proposal exists but target files unreadable or scope not narrow enough | Refined proposal instructions — no product code |
| `implementation_task` | All safety gates pass, scope defined, context readable | Product code change in approved files only |

**Escalation rule:** When uncertain between `measurement_task` and `implementation_task`, choose `measurement_task`. Measurement is always safer than premature implementation.

---

## 5. Product Objective

Vocalype Brain exists to improve Vocalype's core product fundamentals.

### North Star Metric
`successful_dictations_per_active_user_per_week`
Activation event: `first_successful_dictation`

### Product areas to improve (in priority order)
1. **First successful dictation** — activation flow, clear error states, retry capability
2. **Stability** — permissions, crashes, model readiness, clear error messages
3. **Transcription errors** — WER, language accuracy, punctuation, proper nouns
4. **Transcription latency** — capture, preprocessing, model load, inference, paste time
5. **RAM usage** — idle, during transcription, after transcription, model buffers

### Method (never skip steps)
```
measure → diagnose → propose → implement small → test → compare → learn
```

What is not measured cannot be optimized cleanly.

```
backend/
src-tauri/
```

twice — once on raw strings, once on resolved absolute paths.

## 8. Stop Conditions

Stop immediately and report to the founder when any of the following is true.
Do not proceed. Do not guess. Do not try to fix it autonomously.

| # | Condition | Action |
|---|---|---|
| S1 | Repo is dirty with unclear or out-of-scope changes | Stop — clean with `git restore` or ask founder |
| S2 | Task description is vague — approved files or expected output undefined | Stop — ask for clarification |
| S3 | Scope expands during implementation — new files needed beyond approved list | Stop — report and ask for scope approval |
| S4 | Sensitive files required without explicit per-session founder approval | Stop — log refusal, propose alternative |
| S5 | Any check (lint, tests, translation) fails | Stop — report failure, do not commit |
| S6 | Output does not match declared task type | Stop — re-classify before proceeding |
| S7 | Proposed change touches `deriveActivationStatus`, auth reducers, or license validation | Stop — these require separate explicit approval |
| S8 | `allow_product_code_modifications` is `true` in brain config | Stop — safety config anomaly, do not proceed |
| S9 | Implementation model is asked to deploy, delete files, or force-push | Stop — outside allowed scope |
| S10 | The change cannot be rolled back with a single `git checkout -- <file>` | Stop — requires founder review before proceeding |
